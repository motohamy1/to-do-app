# Feature Specification: Add Reminder Screen (Phase 5: Data & State Model)

**Feature Branch**: `001-add-reminder-screen`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: PLAN.md Phase 5: Define what data the screen owns and how it is structured.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Reminder Data Persistence (Priority: P1)

As a user, I want the system to correctly save my title and the precise timestamp I picked so that my reminders trigger at exactly the right time.

**Why this priority**: Correct data and timing.

**Independent Test**: Use Convex Dashboard to look at the `todos` table after saving, and verify the `dueDate` matches the exact Unix timestamp of the date/time picked.

**Acceptance Scenarios**:

1. **Given** the user selects "Apple Juice" for May 12, 11:30 AM, **When** they save, **Then** a new document appears with `text: "Apple Juice"` and a `dueDate` for that precise time.
2. **Given** no project is explicitly selected, **When** saving, **Then** the `projectId` defaults to `null` or a 'General' category.

---

### User Story 2 - Local State Management (Priority: P2)

As a dev, I want the screen to manage its own dirty state while creating so the parent component doesn't need to track partial inputs.

**Why this priority**: Clean architecture and isolation.

**Independent Test**: Mount and unmount the screen with partial data, and ensure unmounting discards the data but doesn't affect the user's primary list until "Save" is called.

**Acceptance Scenarios**:

1. **Given** some text "Clean Desk" is entered, **When** navigating back, **Then** the draft is destroyed and no backend call is initiated.
2. **Given** a successful save, **When** the screen is closed, **Then** the mutation's response is passed back to the list as a successful trigger.

## Data Schema (Reminder Entity)

```typescript
interface Reminder {
  _id: string;        // Generated by Convex
  userId: string;     // From Auth Context
  text: string;       // Non-empty String
  dueDate: number;    // Unix Timestamp (milliseconds)
  status: 'not_started' | 'in_progress' | 'done'; // Defaults to 'not_started'
  projectId?: string; // Optional reference
}
```

## State Ownership

- **Title**: Local `useState<string>` (Screen-local).
- **Date/Time**: Local `useState<Date>` (Screen-local).
- **Backend Sync**: Managed by Convex `useMutation`.
- **Global Context**: Uses `useTheme` for UI tokens and `useTranslation` for text strings.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-501**: System MUST combine date and time selections into a single Unix timestamp for the `dueDate`.
- **FR-502**: System MUST use the user's local timezone for initial picker defaults.
- **FR-503**: System MUST identify the authenticated `userId` before performing the insert mutation.
- **FR-504**: System MUST handle duplicate prevention logic via a local/remote check.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-501**: 100% of created reminders have a `dueDate` in the future.
- **SC-502**: 100% of reminder texts are stored exactly as typed by the user (no forced casing).

## Assumptions

- **Backend**: Uses Convex for real-time data storage and conflict resolution.
- **Timezone**: All dates are stored in UTC but displayed in the user's local timezone.
